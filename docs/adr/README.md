# Журнал архитектурных решений (ADR)

Этот каталог содержит **Architecture Decision Records (ADR)** — короткие документы, фиксирующие ключевые архитектурные решения проекта: *что решили*, *почему* и *какие последствия*.

## Зачем это нужно
- **Объяснимость**: быстро понять, почему система сделана именно так.
- **Воспроизводимость**: однозначно связать поведение системы с версиями конфигов/кода.
- **Снижение техдолга**: решения не «живут в голове», а закреплены в документации.

## Соглашения
- Формат имени: `ADR-XXXX-короткое-имя.md` (XXXX — номер с ведущими нулями).
- Статусы:
  - **Черновик** — документ в работе;
  - **Предложено** — решение предлагается, но ещё не закреплено;
  - **Принято** — решение является актуальным стандартом проекта;
  - **Устарело** — решение заменено новым ADR (в тексте должна быть ссылка на заменяющий ADR).
- Любое изменение «контракта системы» (данные, протоколы, метрики, границы сервисов, модель/рантайм) должно сопровождаться ADR.

## Как добавить новый ADR
1) Создай файл `ADR-XXXX-...md` в этом каталоге.
2) Заполни: **Контекст → Решение → Альтернативы → Последствия → Дальнейшие шаги → Ссылки**.
3) Добавь строку в таблицу выше.
4) Если ADR заменяет старый — пометь старый как **Устарело** и укажи ссылку на новый.
