# ADR-0003: Контракт алертинга (detections→alerts) и метрики пилота

- Статус: Принято
- Дата: 2026-03-04
- Авторы: Максим Яковенко, Михаил Скрыпник, Иван Провков

## Контекст
На ML-уровне модель выдаёт bbox+score по кадрам (детекция). На продуктовом уровне оператору важны не “каждый кадр”, а редкие и понятные подсказки: алерты с временными метками. Качество пилота считается по миссиям метриками TtFC, Recall_event и FP/min.

Важно обеспечить:
- управляемую нагрузку на оператора (FP/min),
- максимизацию полноты по эпизодам (Recall_event),
- измеримое улучшение времени первого контакта (TtFC).

## Решение
1) Продуктовой единицей считать ALERT (событие с временной меткой).
2) Преобразование detections→alerts фиксировать контрактом параметров:
   - conf_thr, W, k, cooldown, tau_gap_end.
3) В пилоте запрещены изменения этих параметров внутри прогона: любая правка = новая версия и новый прогон.
4) Метрики пилота:
   - TtFC: от начала первого GT-эпизода до подтверждения первого алерта, пересёкшегося с эпизодом (±τ),
   - Recall_event: доля найденных GT-эпизодов,
   - FP/min: частота ложных тревог.

## Рассмотренные альтернативы
- Оценка по кадрам (frame-level) вместо алертов.
- Трекинг-объектов как основная продуктовая единица (track-level).
- Один алерт = одна детекция (без окна W и k).

## Последствия
Плюсы:
- Метрики соответствуют реальному UX оператора (алерты как подсказки).
- Можно оптимизировать конфиги “модель+пороги+постобработка” под ограничения FP/min при максимизации Recall_event.

Минусы:
- Появляется чувствительность к параметрам окна и частоте кадров.
- При изменении fps/задержек нужно проверять валидность W/k/cooldown.

## Дальнейшие шаги
- для каждой версии конфига сохранять: параметры, список алертов, TP/FP разметку относительно GT-эпизодов, отчёт по миссии и агрегат по набору миссий.
